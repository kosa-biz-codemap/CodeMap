# CodeCompass 팀 협업 가이드 (Contributing Guidelines)

본 문서는 원활하고 충돌 없는 팀 협업을 위해 약속된 **Git Branch 및 Commit 규칙**을 정의합니다. 모든 팀원은 작업 전 이 규칙을 숙지해 주시기 바랍니다.

---

## 1. Branch 규칙 (GitHub Flow)

본 프로젝트는 불필요한 브랜치를 최소화하고 직관적으로 관리할 수 있는 **GitHub Flow** 기반의 구조를 따릅니다.

### 🛡️ 기본 원칙
* **`main` 브랜치**: 언제든 즉시 배포 가능한 안정적인 코드가 유지되어야 하는 핵심 브랜치입니다. 절대 직접 Push하지 않습니다.
* **작업 브랜치**: `main`에서 브랜치를 분기(checkout)하여 개인 작업을 진행합니다.

### 🏷️ 브랜치 명명 규칙
> **포맷:** `{분류}/{기능명}`

* `feat/login-api` (새로운 기능 개발)
* `fix/header-bug` (버그 수정)
* `docs/readme` (문서 작업)
* `chore/setting` (기타 환경 설정)

---

## 2. Commit 메시지 규칙 (Conventional Commits)

전 세계 개발 표준인 **Angular 커밋 컨벤션**을 따릅니다. 커밋 로그만 보고도 어떤 작업이 이루어졌는지 한눈에 파악할 수 있어야 합니다.

### 🏷️ 커밋 메시지 포맷
> **포맷:** `{태그}: {커밋 내용}`
* 예시: `feat: 로그인 폼 UI 컴포넌트 추가`
* 예시: `fix: JWT 토큰 만료 에러 해결`

### 🛠️ 7대 필수 태그 (Prefix)
1. **`feat:`** 새로운 기능 추가
2. **`fix:`** 치명적인 버그나 에러 수정
3. **`docs:`** README, CONTRIBUTING 등 문서 수정
4. **`style:`** 로직 변화 없이 코드 포맷팅, 세미콜론, 줄바꿈 수정
5. **`refactor:`** 기능은 똑같지만 코드를 더 깔끔하고 효율적으로 리팩토링
6. **`test:`** 테스트 코드 추가 및 수정
7. **`chore:`** 빌드 업무, 패키지 매니저 설정 등 잡다한 작업

---

## 3. PR (Pull Request) 규칙

코드 병합을 위해 PR을 올릴 때, 리뷰어(다른 팀원)가 변경 사항을 쉽게 파악하고 테스트할 수 있도록 템플릿을 제공합니다.
*(참고: 깃허브에서 PR 생성 버튼을 누르면 이 양식이 텍스트 창에 자동으로 입력됩니다.)*

### 📝 PR 작성 시 필수 확인 사항
1. **작업 목적**: 이 코드를 작성한 배경과 이유 (1~2줄 요약)
2. **주요 변경 사항**: 변경된 핵심 로직과 리뷰어가 점검해야 할 체크리스트
3. **리뷰 및 테스트 방법**: 리뷰어가 본인 로컬 환경에서 테스트해 볼 수 있는 명확한 절차 및 명령어 **(필수)**

---

## 4. 작업 흐름 요약 (Golden Workflow)
1. `main`에서 새로운 브랜치를 땁니다. (예: `git checkout -b feat/login`)
2. 커밋 규칙에 맞춰 코드를 작업하고 Commit 합니다.
3. **(핵심) 내 로컬 브랜치에 `main`을 Pull 당겨와서 충돌(Conflict)이 없는지 확인하고, 있다면 내 컴퓨터에서 깨끗이 해결합니다.**
4. 충돌이 완벽히 해결된 코드를 Push 하고 PR을 올립니다.
