## 1. ⚠️ 2차 코드 리뷰 지적 사항에 대한 기술적 해명 (반박)

리뷰어의 제안 중, 현 프로젝트의 설계 사상 및 확장성/비동기 생명주기 관점에서 기존 코드를 고수하는 것이 더 우수하다고 판단하여 반박(유지)한 사항들에 대한 기술적 정당성입니다.

### ① `FETCH_REMOTE_VARS` 플래그의 기본 온보딩 문서 미노출 사유 (#60)
* **리뷰 지적**: `GETTING_STARTED.md` 온보딩 가이드에 `FETCH_REMOTE_VARS` 항목과 설명이 미기재되어 보완 권장.
* **기술적 정당성**: `FETCH_REMOTE_VARS`는 로컬 개발자가 직접 조작할 필요가 없는 **특수 디버그 및 헬퍼 목적의 내부 플래그**입니다. 로컬 기동 가이드에 이를 전면 기재할 경우, 신규 팀원에게 *"로컬 구동 시 깃허브 개인용 토큰(PAT) 주입이 필수적이다"*라는 혼선을 초래할 수 있습니다. 
* **해결 조치**: 문서를 단순하고 명확하게 유지하기 위해 기본 장표에는 노출하지 않고, 고급 세팅 장표에만 주석으로 선별 기술합니다.

### ② `deploy.yml` 내 실제 배포 스크립트 주석 처리 사유 (#60)
* **리뷰 지적**: `deploy.yml` 배포 파이프라인에 실제 배포 명령이 주석 처리된 미완성 상태임.
* **기술적 정당성**: 현재 프로젝트 백엔드의 최종 배포 클라우드 인프라(AWS, GCP, NCP 등) 및 컨테이너 레지스트리가 팀 수준에서 아직 최종 합의되지 않은 CD 설계 뼈대 상태입니다. 실 배포 명령이 비어 있는 채 자동 트리거되면 깃허브 액션 자원 낭비가 심하므로, 트리거를 `workflow_dispatch` (수동 실행)로 임시 변경하여 안전 조치를 취해둔 상태가 진척도 상 타당합니다.

### ③ 예외 테스트 시 `TestClient` 미사용 사유 (#62)
* **리뷰 지적**: FastAPI `TestClient` 가상 네트워크 환경을 구동하여 실제 라우터와 핸들러 연결부를 직접 테스트해야 함.
* **기술적 정당성**: 우리가 구현한 예외 가공 로직(`_build_http_exception_response`)은 사이드 이펙트가 없는 순수 함수(Pure Function) 구조입니다. 굳이 무거운 TestClient를 생성하고 가상 네트워크 포트를 열어 테스트하는 통합 레벨 방식은 **단위 테스트의 핵심 가치(격리성 확보 및 수 밀리초 대의 피드백 속도)**를 훼손합니다. 순수한 예외 인스턴스를 직접 입력하여 입출력을 조밀하게 검증하는 현재 방식이 단위 테스트 설계 표준에 부합합니다.
* **추가 피드백 반영**: 단위 테스트 메소드(`test_http_exception_response_preserves_headers`)의 이름 및 목적과 실제 검증 범위의 혼선을 방지하기 위해, 본 테스트가 핸들러 통과 전체 경로가 아닌 변환 함수의 단위 검증을 목적으로 함을 기술하는 상세 안내 주석(docstring)을 명확하게 보완 조치했습니다.

### ④ `_is_standard_error_response` 통과 시 `data` 필드 정규화(None) 생략 사유 (#62)
* **리뷰 지적**: 표준 에러 응답을 그대로 반환하는 처리 흐름에서 `data` 필드가 강제 `None`으로 검증되지 않아 안전하지 않음.
* **기술적 정당성**: 이미 표준 에러 규격을 갖춘 커스텀 예외의 경우, 디버깅 목적을 위해 **`data` 필드 내부에 상세 메타데이터나 부가 정보를 적재하여 송출**했을 수 있습니다. 이를 핸들러에서 강제로 `None`으로 정제해 지워버릴 경우, 유용한 디버그 정보 유실 및 추적 곤란 등의 심각한 역효과가 나기 때문에 검증 없이 있는 그대로 반환하는 현재 흐름이 타당합니다.

### ⑤ `PreValidateLimit` DTO 3중 중복 인스턴스 생성 유지 사유 (#66)
* **리뷰 지적**: 분기 3곳(`FILE_COUNT_EXCEEDED`, `FILE_SIZE_EXCEEDED`, `REPO_LIMIT_EXCEEDED`)에서 `PreValidateLimit` 객체를 동일 사양(100개, 100KB)으로 중복하여 조립함.
* **기술적 정당성**: 현재는 제한 스펙 임계치가 동일하지만, 추후 비즈니스 정책 변경 시 **각 제한 초과 코드별로 한계 사양은 독립적으로 수정**될 수 있습니다. (예: 파일 수 제한은 200개로 늘리고 크기 제한은 100KB로 유지하는 등). 분기 내부에서 고유의 DTO를 독립 할당하는 방식이 개방-폐쇄 원칙(OCP)과 향후 독립적 요건 확장을 유연하게 수용할 수 있는 확장성 관점에서 더 훌륭한 디자인입니다.

### ⑥ `with` 블록 바깥에서 `.json()` 호출 구조 유지 사유 (#66)
* **리뷰 지적**: `async with httpx.AsyncClient` 블록 종료 후에 `payload = tree_response.json()`을 호출함.
* **기술적 정당성**: `httpx` Response 객체는 비동기 통신을 끝마치고 응답 본문 전체 스트림을 메모리에 이미 로드 완료한 상태입니다. 따라서 커넥션 풀을 가상 풀링 러너에 조기 반납하기 위해 `with` 블록을 즉시 닫은 뒤, 네트워크 소켓 생명주기와 무관한 메모리 상의 파싱(`.json()`)을 바깥에서 진행하는 것이 리소스 효율성 및 관심사 분리 관점에서 올바른 비동기 생명주기 제어 패러다임입니다.
