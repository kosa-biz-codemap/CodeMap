# 💬 Git 커밋 메시지 컨벤션

한 커밋에는 한 가지 문제만 담아 과거 추적이 용이하도록 합니다.

* **커밋 메시지 포맷**:
  ```plain text
  커밋유형: 제목
  (빈 줄)
  본문
  ```
* **작성 규칙**:
  1. 커밋 유형은 대문자로 작성합니다.
  2. 제목과 본문은 한글로 작성합니다.
  3. 제목 끝에는 마침표(`.`)를 붙이지 않습니다.
  4. 제목은 영문 기준 50자 이내로 작성합니다.
  5. 본문에는 변경 내용과 이유를 '무엇'과 '왜' 위주로 작성하며, 글머리 기호(`-`)를 사용합니다.

* **커밋 유형**:
  * `FEAT`: 새로운 기능 추가
  * `FIX`: 버그 수정
  * `DOCS`: 문서 수정
  * `STYLE`: 코드 formatting, 세미콜론 누락, 코드 자체의 변경이 없는 경우
  * `REFACTOR`: 코드 리팩토링
  * `TEST`: 테스트 코드 추가
  * `CHORE`: 패키지 매니저 수정, 그 외 기타 수정 (.gitignore 등)
  * `DESIGN`: CSS 등 사용자 UI 디자인 변경
  * `COMMENT`: 주석 추가 및 변경
  * `RENAME`: 파일/폴더명 수정 또는 이동
  * `REMOVE`: 파일 삭제
  * `!BREAKING CHANGE`: 커다란 API 변경
  * `!HOTFIX`: 치명적인 버그 긴급 수정

# 🐍 Python PEP 8 컨벤션

### 💻 코드 컨벤션 (Code Convention)
* **들여쓰기**: 탭(Tab) 대신 스페이스(Space) 4칸을 사용합니다.
* **최대 줄 길이**: 모든 줄은 최대 79자로 제한하고, 독스트링 및 주석은 72자로 제한합니다.
* **빈 줄 (Blank Lines)**:
  * 최상위 함수와 클래스 정의 사이에는 두 줄(2 lines)을 띄웁니다.
  * 클래스 내부의 메서드 정의 사이에는 한 줄(1 line)을 띄웁니다.
* **Imports**:
  * `import`문은 파일의 맨 위에, 항상 별도의 줄에 작성합니다.
  * 표준 라이브러리, 서드파티, 로컬 모듈 순서로 그룹화합니다.
* **띄어쓰기 (Whitespace)**:
  * 괄호 안이나 쉼표 바로 앞에는 공백을 넣지 않습니다.
  * 대입 연산자(`=`), 비교 연산자(`==`, `<`, `>`) 주변에는 스페이스를 한 칸씩 둡니다.

### 📝 명명 규칙 (Naming Convention)
* **클래스명**: `CapWords` (PascalCase) 방식을 사용합니다. (예: `MyClass`, `UserModel`)
* **함수 및 변수명**: 소문자와 언더스코어(`_`)를 사용하는 `snake_case` 방식을 사용합니다. (예: `def calculate_total():`, `user_id = 1`)
* **상수**: 모든 글자를 대문자로 하고 언더스코어로 구분하는 `UPPER_SNAKE_CASE`를 사용합니다. (예: `MAX_OVERFLOW`, `TOTAL_COUNT`)
* **인스턴스/클래스 메서드**: 첫 번째 인자는 항상 `self` (인스턴스), `cls` (클래스)를 사용합니다.
* **비공개 속성/메서드**: 내부적으로만 쓰이는 변수나 메서드는 언더스코어 한 개(`_`)로 시작합니다.

# 🌿 브랜치 규칙 (Branch Convention)

GitHub Flow 기반의 구조를 따르며, 목적에 맞게 브랜치를 분기하여 작업합니다.
* `main`: 언제든 즉시 배포 가능한 안정적인 코드가 유지되는 브랜치 (직접 Push 금지)
* **브랜치 명명 규칙**: `{분류}/{기능명}` 포맷을 사용합니다.
  * `feat/login-api` (새로운 기능 개발)
  * `fix/header-bug` (버그 수정)
  * `docs/readme` (문서 작업)
  * `chore/setting` (기타 환경 설정)

# 🚀 PR 규칙 (Pull Request Convention)

* **PR 제목**: `{커밋 유형}: {PR 제목}` (예: `FEAT: 로그인 폼 UI 컴포넌트 추가`)
* **PR 내용 작성 필수 항목**:
  1. **작업 목적**: 코드 작성 배경과 이유
  2. **주요 변경 사항**: 변경된 핵심 로직과 리뷰어 체크리스트
  3. **리뷰 및 테스트 방법**: 리뷰어가 로컬에서 테스트해 볼 수 있는 절차 및 명령어
* **작업 흐름**:
  1. `main` 브랜치에서 최신 버전을 Pull 받은 후 새로운 브랜치 생성
  2. 작업 완료 후 로컬 `main`을 머지하여 충돌(Conflict) 해결
  3. 충돌 없는 코드를 원격 저장소에 Push 후 PR 생성
