# CodeMap 제품 포지셔닝 및 적용 전략 v3

> 문서 상태: v2 리뷰를 반영한 v3 초안
>
> 핵심 질문: "코딩 에이전트가 이미 레포를 잘 읽는 시대에, CodeMap은 왜 필요한가?"
>
> 결론: CodeMap의 경쟁력은 코사인 유사도나 임베딩 자체가 아니라, **낯선 코드베이스에서 발생하는 "이해의 비대칭"을 팀이 함께 해소하는 협업 온보딩 플랫폼 경험**에 있다.

---

## 1. 상위 미션: 이해의 비대칭 해소

코드베이스의 정보는 레포 안에 다 있다. 문제는 정보가 없는 것이 아니라 **같은 정보를 보고도 사람마다 이해의 깊이와 속도가 다른 것**이다.

- **같은 레포를 보고도** 시니어는 3분 만에 핵심을 파악하고 신입은 3시간 헤맨다.
- **같은 레포를 보고도** 개발자는 구조를 이해하지만 PM은 이해하지 못한다.
- **같은 레포를 보고도** 어제 합류한 사람과 1년 전부터 있던 사람의 이해 수준이 다르다.
- **같은 레포에 질문해도** 좋은 질문을 만들 수 있는 사람과 무엇을 물어야 할지 모르는 사람의 결과가 다르다.

CodeMap은 이 "이해의 비대칭"을 해소하는 것을 상위 미션으로 삼는다. 모든 기능과 USP는 이 미션 아래에서 정당화된다.

```text
USP 1 (Question Zero)       → 질문 능력에 따른 이해 격차 해소
USP 2 (역할별 맵)            → 역할에 따른 이해 격차 해소
USP 5 (공용 컨텍스트)        → 시점에 따른 이해 격차 해소
USP 9 (Team Onboarding Space) → 경험에 따른 이해 격차 해소 (시니어↔신입)
```

## 2. 핵심 결론

CodeMap은 "레포지토리에 질문하는 또 하나의 AI 챗봇"으로 포지셔닝하면 Codex, Claude Code, Antigravity 같은 범용 코딩 에이전트와 정면으로 겹친다. 이 경쟁에서는 단순 RAG, 임베딩, pgvector 사용만으로 차별화하기 어렵다.

CodeMap이 가져가야 할 자리는 다음과 같다.

> **CodeMap은 GitHub 레포지토리를 분석하여, 팀원 모두가 같은 프로젝트를 역할별로 이해하고 함께 온보딩하는 협업 플랫폼이다. 시니어의 반복 설명을 줄이고, 신입의 막막함을 없애며, 비개발 협업자도 코드 구조를 자기 관점에서 이해할 수 있게 한다.**

핵심은 네 가지다.

1. **질문하기 전에 필요한 것을 먼저 보여준다.**
   - 사용자가 좋은 질문을 만들 능력이 없어도 시작할 수 있어야 한다.
   - 실행 방법, 진입점, 핵심 파일, 주요 흐름, 위험 요소를 선제적으로 구성한다.

2. **같은 레포를 역할과 목적에 따라 다르게 설명한다.**
   - 개발자에게는 실행·의존성·수정 시작점을 보여준다.
   - PM에게는 기능 경계·API·데이터 흐름을 보여준다.
   - 디자이너에게는 화면·컴포넌트·상태·디자인 자산의 연결을 보여준다.
   - QA에게는 주요 사용자 흐름·예외 처리·테스트 지점을 보여준다.

3. **일회성 답변이 아니라 근거가 남는 공용 결과물을 만든다.**
   - 파일 경로와 코드 근거가 연결된 리포트를 생성한다.
   - 팀원이 같은 기준점에서 프로젝트를 이해하도록 한다.
   - 분석 시점과 커밋을 기록하고, 이후에는 변경된 부분을 갱신할 수 있어야 한다.

4. **이해 과정 자체를 팀과 공유한다.**
   - 혼자 AI에게 질문하고 끝나는 것이 아니라, 팀 공유 대화방에서 질문과 답변이 축적된다.
   - 시니어가 답변을 보충하고, 검증된 Q&A가 온보딩 문서로 승격된다.
   - 신입은 부끄러운 질문을 개인 대화방에서 먼저 해보고, 유용한 것만 공유할 수 있다.

따라서 `FastAPI + LangChain + 임베딩 + pgvector`는 제품의 USP가 아니라 이 경험을 구현하는 기술 수단이다. 기술적 차별화는 특정 라이브러리 이름이 아니라 **구조 분석, 정확 검색, 의미 검색, 에이전트 탐색, 근거 검증을 질문별로 조합하는 파이프라인**에서 만들어야 한다.

## 3. 출발점이 된 기술적 의문에 대한 답

### 3.1 첫 프로젝트에서는 코사인 유사도를 쓸 일이 없는가?

반은 맞고 반은 다르다.

코사인 유사도 검색은 "여러 프로젝트가 DB에 많이 쌓여야" 사용할 수 있는 기능이 아니다. 첫 번째 레포 하나만 등록해도 사용자의 질문 벡터와 **그 레포 내부의 코드 청크 벡터들**을 비교할 수 있다. 즉, 비교 대상은 다른 프로젝트가 아니라 같은 프로젝트 안의 함수, 클래스, 설정, 문서 청크다.

다만 첫 분석부터 모든 파일을 무조건 임베딩하는 것이 항상 효율적인 것은 아니다.

- 작은 레포를 한 번만 분석한다면 인덱싱 비용이 오히려 더 클 수 있다.
- `package.json`, `.env.example`, `main.py`, 라우터 목록처럼 위치와 형식이 명확한 정보는 정적 분석과 정확 검색이 더 빠르고 정확하다.
- 임베딩은 "인증과 의미상 관련된 코드가 어디에 흩어져 있는가?"처럼 표현이 다양하거나 범위가 넓은 질문에서 보완 가치가 커진다.
- 같은 레포에서 질문과 문서 생성이 반복될수록 한 번 만든 인덱스를 재사용할 수 있어 효율이 높아진다.

따라서 올바른 결론은 "첫 프로젝트에는 벡터 검색이 필요 없다"가 아니라 다음에 가깝다.

> **첫 분석에서도 벡터 검색은 사용할 수 있지만, 레포 크기·질문 종류·재사용 가능성에 따라 선택적으로 사용해야 한다.**

### 3.2 RAG는 PDF 같은 외부 정보를 모델에 주는 기술 아닌가?

PDF는 흔한 예시일 뿐이다. RAG의 본질은 모델이 원래 알지 못하거나 현재 프롬프트에 들어 있지 않은 자료에서 **질문에 필요한 근거만 찾아 컨텍스트로 제공하는 것**이다. 그 자료는 PDF일 수도 있고, 사내 문서나 데이터베이스일 수도 있으며, 코드 레포지토리일 수도 있다.

레포를 서버에 클론했다고 해서 모델이 모든 파일을 자동으로 알고 있는 것은 아니다. 모델이 파일을 이해하려면 다음 중 하나가 필요하다.

- 레포 전체 또는 일부를 프롬프트에 직접 넣는다.
- 모델에게 `tree`, `grep`, 파일 읽기 같은 도구를 주고 필요한 파일을 찾게 한다.
- 미리 만든 검색 인덱스에서 관련 청크를 가져온다.
- 위 방법을 함께 사용한다.

작은 레포는 전체 파일을 읽혀도 되지만, 큰 레포는 토큰 한도·비용·지연 시간 때문에 매 질문마다 전체 코드를 넣는 방식이 비효율적이다. 검색은 모델이 읽어야 할 후보를 줄여 **입력 토큰과 탐색 시간을 절약**할 수 있다. 반대로 검색 품질이 낮으면 중요한 주변 문맥을 잃을 수 있으므로, 검색 결과만 읽고 답하게 해서는 안 된다.

### 3.3 범용 코딩 에이전트도 결국 키워드 검색과 파일 읽기를 하지 않는가?

그렇다. 좋은 코딩 에이전트는 사용자가 "인증 로직은 어디에 있나요?"라고 물었을 때 `인증`이라는 한글 단어만 찾지 않는다. 프로젝트 구조를 보고 `auth`, `token`, `JWT`, `OAuth`, `session`, `current_user`, `guard`, `password_hash` 같은 후보를 만들고, 관련 파일을 읽고, 실패하면 다른 경로를 탐색한다.

이 방식은 특히 다음 상황에서 강하다.

- 로컬 레포에 즉시 접근할 수 있다.
- 파일을 직접 열어 주변 문맥을 온전히 읽을 수 있다.
- 검색이 실패했을 때 스스로 검색어와 탐색 경로를 바꿀 수 있다.
- 코드 실행, 테스트, git 이력 확인까지 연결할 수 있다.

따라서 CodeMap이 "벡터 검색이라서 범용 코딩 에이전트보다 더 정확하다"고 주장해서는 안 된다. 임베딩은 만능 대체재가 아니라 **에이전트가 넓은 레포에서 탐색 후보를 빠르게 만드는 검색 도구 중 하나**다.

### 3.4 서버에 올리는 방식이 로컬 도구보다 비효율적일 수 있는가?

그럴 수 있다. 특히 한 명의 숙련 개발자가 작은 레포를 한 번 질문하는 상황이라면 로컬 코딩 에이전트가 더 빠르고 자연스럽다. 서버 방식에는 클론·업로드, 필터링, 파싱, 인덱싱, 저장, 권한 관리 비용이 추가된다.

CodeMap은 이 약점을 숨기지 말고, 서버 분석이 가치 있는 조건에 집중해야 한다.

- 여러 사람이 같은 분석 결과를 재사용한다.
- 비개발자도 별도 환경 설정 없이 접근해야 한다.
- 표준화된 온보딩 리포트가 반복적으로 필요하다.
- 같은 커밋을 기준으로 팀이 공통 근거를 봐야 한다.
- 여러 질문과 역할별 뷰에서 분석 인덱스를 재사용한다.
- 시간이 지난 뒤 변경점을 비교하고 문서를 갱신한다.

민감한 사설 레포에 대해서는 서버 업로드가 오히려 도입 장벽이 된다. 제품화 단계에서는 저장 기간, 삭제 정책, 암호화, 접근 제어를 명시하고 장기적으로는 온프레미스·자가 호스팅·로컬 인덱싱 같은 선택지를 검토해야 한다.

## 4. 문제 정의: README가 있어도 온보딩이 끝나지 않는 이유

README의 문제를 단순히 "부실할 수 있다"로만 설명하면 약하다. README는 중요하지만, 애초에 하나의 파일이 모든 온보딩 요구를 충족하도록 설계된 것은 아니다.

### 4.1 README의 구조적 한계

1. **작성 목적이 프로젝트마다 다르다.**
   - 라이브러리는 설치법과 API 사용법에 집중한다.
   - 제품 레포는 서비스 소개와 배포 방법에 집중한다.
   - 수업·토이 프로젝트는 제출 형식이나 결과 화면에 집중할 수 있다.
   - 모노레포는 루트 README와 하위 패키지 문서가 분리될 수 있다.

2. **작성 방식과 품질 기준이 통일되어 있지 않다.**
   - 같은 팀 안에서도 작성자에 따라 깊이와 용어가 다르다.
   - 실행 명령만 있고 아키텍처 설명이 없거나, 반대인 경우도 있다.
   - 핵심 정보가 README, Wiki, Notion, 이슈, 코드 주석에 나뉘어 있을 수 있다.

3. **코드와 자동으로 동기화되지 않는다.**
   - 폴더 구조, 환경변수, 실행 명령, API 경로가 바뀌어도 README는 그대로 남을 수 있다.
   - 문서의 마지막 수정일만으로 실제 코드와 일치하는지 판단하기 어렵다.

4. **역할별 질문에 답하지 못한다.**
   - 신규 백엔드 개발자와 디자이너가 알고 싶은 내용은 다르다.
   - 동일한 README 한 장으로는 역할별 깊이와 용어를 조절하기 어렵다.

5. **'무엇을 먼저 읽을지'까지 안내하지 않는 경우가 많다.**
   - 파일 목록은 있어도 핵심 진입점과 호출 흐름이 설명되지 않을 수 있다.
   - 코드가 실제로 어떤 순서로 실행되는지, 변경하려면 어디서 시작해야 하는지 빠져 있을 수 있다.

6. **근거와 불확실성이 분리되지 않는다.**
   - 문서에 적힌 실행법이 현재도 유효한지 확인하기 어렵다.
   - 코드에서 확인된 사실, README에서만 주장하는 내용, AI가 추론한 내용을 구분하기 어렵다.

7. **README는 "혼자 읽는 문서"이다.**
   - 읽다가 이해가 안 되면 물어볼 사람을 따로 찾아야 한다.
   - 내가 이해한 내용이 맞는지 확인받을 방법이 없다.
   - 같은 README를 읽은 팀원들이 같은 수준으로 이해했는지 알 수 없다.

### 4.2 사용자가 실제로 겪는 문제

- "설치는 했는데 어느 서버를 먼저 실행해야 하지?"
- "`.env`에 어떤 키가 필요한지 한곳에 정리되어 있나?"
- "로그인 화면에서 DB까지 데이터가 어떻게 흐르지?"
- "이 파일이 중요해 보이는데 실제로 어디서 호출되지?"
- "내가 프론트엔드 담당인데 백엔드 문서를 어디까지 읽어야 하지?"
- "README와 실제 코드가 다른데 무엇을 믿어야 하지?"
- "질문할 사람도 없고, AI에게 무엇부터 물어봐야 할지도 모르겠다."
- "이렇게 이해한 게 맞는 건지 확인받고 싶은데 누구한테 물어보지?"
- "시니어한테 물어보기엔 너무 기초적인 질문 같아서 부담스럽다."

CodeMap이 해결할 핵심 문제는 README의 존재 여부가 아니라 다음의 간극이다.

> **레포에는 정보가 존재하지만, 처음 온 사람이 자기 역할과 목적에 맞게 읽을 수 있는 경로와 검증된 설명이 없고, 이해 과정을 팀과 공유할 방법도 없다.**

## 5. 타겟 사용자 재정의

기존 초안의 "신입/주니어 개발자"와 "팀 프로젝트에 새로 합류한 팀원"은 서로 겹친다. 전자는 숙련도이고 후자는 상황이기 때문이다. v3에서는 타겟을 **사용 상황, 핵심 사용자, 도입 결정자, 결과 소비자**로 나눈다.

### 5.1 핵심 사용 상황

> **낯선 레포를 맡았고, 짧은 시간 안에 구조를 이해해 다음 행동을 결정해야 하는 상황**

여기에는 다음 사례가 포함된다.

- 신입 개발자가 첫 팀 프로젝트에 합류한다.
- 경력 개발자가 다른 팀 또는 레거시 프로젝트로 이동한다.
- 오픈소스 기여자가 처음 이슈를 맡는다.
- 외주·협업 개발자가 기존 레포를 인수한다.
- 학생이 참고용 레포를 학습한다.

즉, 핵심 타겟은 단순히 "주니어"가 아니라 **낯선 코드베이스의 신규 진입자**다. 주니어에게 문제가 더 크지만, 숙련 개발자도 큰 레거시 레포나 낯선 기술 스택 앞에서는 같은 문제를 겪는다.

### 5.2 사용자 및 이해관계자 구분

| 구분 | 대표 사용자 | 핵심 목표 | CodeMap이 제공해야 할 가치 |
|---|---|---|---|
| 핵심 사용자 | 신규 합류 개발자, 프로젝트 이동 개발자, 오픈소스 기여자 | 실행하고, 구조를 이해하고, 첫 수정 지점을 찾기 | 개발자용 온보딩 맵, 읽을 순서, 실행 체크리스트, 근거 기반 Q&A |
| 도입 결정자 | 팀장, 테크리드, 교육자, 프로젝트 관리자 | 반복 설명을 줄이고 온보딩 품질을 표준화하기 | 재사용 가능한 리포트, 역할별 가이드, 최신성 확인, 공유 링크, 온보딩 타임라인 |
| 협업 소비자 | PM, 디자이너, QA, 기획자 | 자신의 업무와 코드 구조의 접점을 이해하기 | 역할별 용어 번역, 기능·화면·API·데이터 흐름 뷰 |
| 개인 탐색 사용자 | 학생, 사이드 프로젝트 개발자, 레퍼런스 조사자 | 낯선 구현을 빠르게 학습하거나 비교하기 | 학습 경로, 아키텍처 요약, 참고 포인트, 질문 템플릿 |

### 5.3 우선순위

MVP의 1차 타겟은 넓게 잡지 않는다.

1. **1차:** 팀 프로젝트에 새로 합류한 개발자
2. **2차:** 그 개발자를 온보딩해야 하는 팀장·테크리드
3. **확장:** PM·디자이너·QA용 역할별 프로젝트 맵
4. **추가 활용:** 개인 학습, 오픈소스 탐색, 레퍼런스 분석

비개발 협업자까지 처음부터 모두 만족시키려 하면 결과가 얕아질 수 있다. 먼저 개발자 온보딩의 정확도와 근거성을 만들고, 동일한 분석 데이터 위에 역할별 뷰를 확장하는 편이 현실적이다.

## 6. 경쟁 구도: 4축 비교

Codex, Claude Code, Antigravity는 코드 수정만 하는 단순 도구가 아니다. 코드베이스 탐색, 질문 응답, 문서 생성, 원격·클라우드 작업, 여러 도구 연결까지 계속 확장되고 있다. PM과 디자이너도 이러한 도구를 사용할 수 있으므로 "비개발자는 CLI를 못 쓴다"만으로는 지속 가능한 차별점이 되지 못한다.

### 6.1 4축 경쟁 비교

| 구분 | 범용 코딩 에이전트 | VS Code + 확장 | Notion / Wiki | **CodeMap** |
|---|---|---|---|---|
| 코드 이해 | ★★★★★ | ★★★ | ★ | ★★★★ |
| 코드 수정 | ★★★★★ | ★★★★★ | ✗ | ✗ |
| 비개발자 접근성 | ★★ | ★ | ★★★★★ | ★★★★ |
| 역할별 맞춤 | ✗ | ✗ | 수동 작성 | **자동 생성** |
| 근거 기반 설명 | 세션 내 | 파일 수준 | 작성자 의존 | **파일+줄+검증 상태** |
| 팀 실시간 공유 | ✗ | Live Share (편집) | ★★★★★ | **★★★★** |
| 최신성 관리 | 세션 내 | ✗ | 수동 | **커밋 기반 자동** |
| 온보딩 특화 | ✗ | ✗ | 수동 | **핵심 목적** |

이 테이블로 보면 CodeMap은 **"Notion의 협업 + 코딩 에이전트의 코드 이해 + 온보딩 특화"의 교집합**이라는 포지션이 명확해진다.

### 6.2 기존 비교표 (상세)

| 구분 | 범용 코딩 에이전트 | CodeMap이 집중할 영역 |
|---|---|---|
| 핵심 목적 | 이해한 뒤 코드를 수정·실행·검증하는 작업 완수 | 낯선 레포를 팀이 함께 이해하고 온보딩하는 과정 |
| 상호작용 시작점 | 사용자가 과업이나 질문을 제시 | 사전 분석된 프로젝트 맵과 역할별 학습 경로를 먼저 제시 |
| 탐색 방식 | 매 세션 에이전트가 필요한 파일을 능동 탐색 | 구조화 분석 결과를 저장하고 여러 사용자·질문에서 재사용 |
| 기본 결과 | 답변, 코드 변경, 실행 결과, PR 등 | 근거 연결형 온보딩 리포트, 역할별 뷰, 체크리스트 |
| 사용 단위 | 개인 또는 에이전트 작업 세션 | 프로젝트·커밋·팀을 기준으로 한 지속 가능한 지식 자산 |
| 협업 | 1인 세션, 결과를 수동으로 공유 | 팀스페이스에서 이해 과정 자체를 공유 |
| 성공 기준 | 요청한 개발 작업의 정확한 완료 | 첫 실행 및 첫 기여까지 걸리는 시간과 반복 설명 비용 감소 |

이 표는 기능의 절대적 배타성을 주장하지 않는다. 범용 에이전트도 문서를 만들 수 있고 CodeMap도 Q&A를 제공할 수 있다. 차이는 "할 수 있느냐"보다 **어떤 과정을 기본 제품 경험으로 표준화하고 축적하느냐**에 있다.

### 6.3 CodeMap이 굳이 이기려 하지 않아도 되는 영역

- 자유로운 코드 수정과 대규모 리팩토링
- 로컬 명령 실행과 디버깅
- 테스트 작성 및 PR 생성
- 개발자 개인의 범용 에이전트 작업
- 모든 종류의 개발 질문에 대한 최강 모델 경쟁

### 6.4 함께 사용할 수 있는 포지션

CodeMap은 코딩 에이전트의 대체재가 아니라 앞단의 공통 이해 레이어가 될 수 있다.

```text
CodeMap으로 프로젝트 맥락과 읽을 경로를 익힌다
→ Codex / Claude Code / Antigravity로 실제 수정 작업을 수행한다
→ 변경된 커밋을 CodeMap에서 다시 분석해 팀 문서를 갱신한다
```

장기적으로는 CodeMap의 구조화된 프로젝트 맵을 Markdown, API 또는 MCP 형태로 내보내 범용 에이전트가 더 좋은 초기 컨텍스트로 사용하게 할 수도 있다.

## 7. CodeMap의 확장된 USP

### USP 1. 질문이 없어도 시작되는 'Question Zero' 온보딩

> 해소하는 비대칭: **질문 능력에 따른 이해 격차**

범용 챗봇은 좋은 질문을 받을수록 좋아진다. CodeMap은 질문 이전의 막막함을 해결한다.

- 이 프로젝트는 무엇인가?
- 어떻게 실행하는가?
- 어디서 시작되는가?
- 어떤 파일을 어떤 순서로 읽어야 하는가?
- 첫 작업 전에 무엇을 조심해야 하는가?

이 질문들을 사용자가 입력하지 않아도 기본 리포트에 포함한다.

### USP 2. 같은 레포를 역할별로 번역하는 프로젝트 맵

> 해소하는 비대칭: **역할에 따른 이해 격차**

하나의 일반 요약을 모두에게 보여주는 대신 같은 분석 근거를 다른 관점으로 재구성한다.

| 역할 | 우선 제공 내용 |
|---|---|
| 프론트엔드 개발자 | 페이지 진입점, 라우팅, 상태 관리, API 호출, 공통 컴포넌트, 디자인 토큰 |
| 백엔드 개발자 | 앱 진입점, 라우터, 서비스, DB 모델·연결, 인증, 예외 처리 |
| 풀스택 개발자 | 프론트 요청부터 백엔드·DB까지의 종단 흐름, 실행 순서 |
| PM·기획자 | 사용자 기능, 도메인 경계, 화면과 API의 연결, 주요 정책 |
| 디자이너 | 화면·레이아웃·컴포넌트 계층, 상태별 UI, 디자인 자산 위치 |
| QA | 핵심 사용자 시나리오, API·화면 접점, 오류 처리, 테스트 파일과 미검증 영역 |

역할 맞춤화는 말투만 바꾸는 것이 아니라 **검색 대상, 정보 우선순위, 설명 깊이, 결과 템플릿**을 바꾸는 것이다.

### USP 3. 근거와 불확실성이 보이는 설명

모든 중요한 주장에 다음 정보를 연결한다.

- 분석한 커밋 SHA와 시각
- 파일 경로와 가능하면 줄 범위
- 코드에서 확인한 사실인지, 문서에서 가져온 내용인지, 모델의 추론인지
- 신뢰도 또는 추가 확인 필요 표시
- README와 실제 설정이 충돌할 때의 경고

예시:

```text
실행 명령: pnpm dev
근거: frontend/package.json > scripts.dev
상태: 코드에서 확인됨

필수 환경변수: DATABASE_URL
근거: backend/app/infra/config.py
상태: 코드에서 확인됨

권장 실행 순서: DB → Backend → Frontend
근거: docker-compose.yml, 각 실행 스크립트
상태: 여러 파일을 바탕으로 추론됨 — 실제 실행 검증 필요
```

### USP 4. 구조·관계·의미를 함께 쓰는 하이브리드 코드 이해

CodeMap은 벡터 DB 하나에 의존하지 않는다.

- 파일 트리와 설정 파일 파싱
- AST 기반 심볼·함수·클래스 추출
- import·호출·라우트·DB 관계 그래프
- 정확 키워드 및 정규식 검색
- BM25 또는 전문 검색
- 임베딩 기반 의미 검색
- 에이전트의 반복 탐색과 원문 파일 확인

질문에 맞는 검색 방식을 라우팅한 뒤, 마지막에는 원문 파일과 주변 문맥을 읽어 답을 검증한다. 이 결합이 LangChain이나 pgvector라는 개별 기술보다 중요한 구현 경쟁력이다.

### USP 5. 한 번 분석하고 팀 전체가 반복 사용하는 공용 컨텍스트

> 해소하는 비대칭: **시점에 따른 이해 격차**

로컬 에이전트가 세션마다 다시 탐색할 수 있는 것과 달리 CodeMap은 다음을 프로젝트 자산으로 축적한다.

- 구조화된 파일·심볼 메타데이터
- 역할별 온보딩 리포트
- 자주 묻는 질문과 검증된 근거
- 분석 커밋과 최신성 상태
- 변경 후 영향받은 문서 섹션

이 USP는 사용자가 한 명이고 질문이 한 번뿐일 때는 약하다. 팀원 수, 반복 질문, 레포 변경이 늘수록 강해진다.

### USP 6. '읽기'에서 '첫 행동'까지 연결하는 온보딩 경로

리포트가 설명으로 끝나지 않고 사용자가 다음 행동을 선택할 수 있게 한다.

- 로컬 실행 체크리스트
- 역할별 추천 파일 5개와 추천 이유
- 첫 번째로 따라가 볼 기능 흐름
- 쉬운 첫 기여 후보 또는 학습 과제
- 확인이 필요한 환경변수와 담당자 질문 목록
- 완료 표시가 가능한 온보딩 체크리스트

### USP 7. 문서의 최신성과 변경 영향을 관리하는 살아 있는 온보딩

초기 MVP 이후 가장 강한 확장 방향이다.

- 분석 기준 커밋 표시
- 새 커밋 또는 PR 이후 변경된 모듈 감지
- 기존 리포트에서 오래된 섹션 표시
- 인증·DB·API 등 핵심 흐름 변경 알림
- 전체 재분석이 아닌 변경 파일 중심 증분 갱신

단발성 AI 요약보다 "현재 코드와 얼마나 일치하는가"를 관리하는 제품이 팀 문서로서 더 큰 가치를 가진다.

### USP 8. 공유와 접근성을 기본값으로 제공

- 별도 IDE나 로컬 개발환경 없이 웹에서 접근
- 권한이 있는 팀원에게 동일한 프로젝트 뷰 제공
- Markdown·PDF·공유 링크 내보내기
- 역할별 링크 또는 뷰 저장
- 기술 용어 설명과 깊이 조절

단, "웹이라서 쉽다"는 단독 USP가 아니라 앞선 구조화·역할화·근거화 기능을 전달하는 UX 장점으로 봐야 한다.

### USP 9. Team Onboarding Space — 이해 과정을 팀과 공유하는 협업 공간 (신규)

> 해소하는 비대칭: **경험에 따른 이해 격차 (시니어↔신입)**

기존의 Codex, Claude Code, Antigravity는 모두 **1인 세션** 기반이다. 대화를 팀원과 실시간으로 공유하는 기능은 없다. VS Code Live Share도 코드 편집에 초점이지, "프로젝트 이해를 함께 하는 공간"은 아니다.

CodeMap은 코드 이해 과정 자체를 팀 협업으로 만든다.

#### 핵심 기능

```text
프로젝트 등록
→ 팀스페이스 생성 (Workspace)
→ 팀원 초대 (이메일, 링크)
→ 공유 대화방 (Shared Q&A)
   - 누구든 질문하면 팀 전체가 볼 수 있음
   - 답변에 달린 근거 파일·코드를 함께 확인
   - 시니어가 댓글이나 반응으로 "이건 맞아" / "여기도 봐봐" 보충 가능
→ 개인 대화방 (Private Q&A)
   - "바보 같은 질문"을 혼자 먼저 해볼 수 있음
   - 유용한 질문을 발견하면 공유방으로 옮길 수 있음
→ 팀 온보딩 리포트
   - 공유 대화에서 검증된 질문·답변이 축적되면 온보딩 문서로 승격
```

#### 왜 강력한 USP인가

| 관점 | 설명 |
|---|---|
| 경쟁 차별화 | 코딩 에이전트는 1인 세션, Notion/Wiki는 수동 작성. "AI 코드 분석 + 팀 협업"의 교집합은 비어 있다. |
| 페인포인트 일치 | 신입이 혼자 AI에게 질문하면 "내가 제대로 이해한 건지" 확신이 없다. 시니어가 같은 화면에서 보고 "맞아, 거기가 핵심이야"라고 한마디 하면 온보딩 시간이 극적으로 줄어든다. |
| 노션 멘탈 모델 | 개발자든 비개발자든 이미 "팀스페이스에 초대 → 같은 문서를 보며 작업"하는 경험에 익숙하다. |
| 도입 결정자 설득 | 팀장·테크리드 입장에서 "온보딩 시간이 단축된다"는 말보다 "팀원들이 뭘 이해하고 뭘 모르는지 볼 수 있다"가 더 구체적인 가치다. |

#### 추가 기능 아이디어

- **"이 답변이 도움이 됐어요" 반응**: 유용한 Q&A가 리포트 상단에 올라감
- **역할별 필터**: PM이 공유방에서 "PM 관점 질문"만 모아 볼 수 있음
- **온보딩 진행률**: 팀원별로 추천 파일 확인, 체크리스트 진행률 표시
- **"이 질문은 공개/비공개" 토글**: 대화 입력 시 간단한 토글로 선택
- **알림**: "OOO님이 새 질문을 했습니다" → 시니어가 필요할 때 들어와서 답변 보충

#### 단계적 구현

| 단계 | 범위 | 기술 |
|---|---|---|
| **Phase 1 (MVP)** | 비동기 공유 — 질문·답변이 팀스페이스에 축적, 페이지 새로고침으로 확인 | DB 기반 CRUD, 권한 관리 |
| **Phase 2** | 준실시간 알림 — 새 질문/답변 시 알림, 폴링 또는 SSE | Server-Sent Events |
| **Phase 3** | 실시간 동기화 — 같은 화면에서 커서·타이핑·답변이 보임 | WebSocket, 상태 동기화 |

Phase 1만으로도 Codex/Claude Code와 완전히 다른 협업 경험을 제공할 수 있다.

## 8. RAG·임베딩·코사인 유사도의 정확한 제품 내 역할

### 8.1 기술은 기능이 아니라 검색 전략이다

사용자는 "코사인 유사도 검색"을 사기 위해 CodeMap을 쓰지 않는다. 사용자가 원하는 것은 정확하고 빠른 프로젝트 이해다. 따라서 화면과 발표에서 기술을 앞세우기보다 어떤 질문을 더 잘 해결하는지 설명해야 한다.

### 8.2 질문별 검색 라우팅

| 질문 또는 생성 항목 | 1차 방식 | 보조 방식 | 임베딩 우선도 |
|---|---|---|---|
| 어디서 실행되나요? | manifest·스크립트·Docker 설정 파싱 | README 대조, 에이전트 검증 | 낮음 |
| 환경변수는 어디에 정의되나요? | 설정 파일·환경변수 패턴 검색 | import·참조 관계 추적 | 낮음 |
| API 라우터 구조를 설명해줘 | 프레임워크별 정적 분석·AST | 파일 읽기 및 그래프 구성 | 낮음~중간 |
| DB 연결은 어디서 이루어지나요? | 설정·클라이언트 생성 패턴 검색 | import/call graph, 의미 검색 | 중간 |
| 인증 로직은 어디에 있나요? | 키워드 확장 + 심볼·의존성 검색 | 벡터 검색 + 에이전트 원문 확인 | 중간~높음 |
| 처음 읽어야 할 파일 5개는? | 진입점·fan-in·의존성·역할 점수 | 파일 요약 임베딩, LLM 순위화 | 중간 |
| 결제 요청 흐름을 설명해줘 | 라우트·호출·데이터 관계 그래프 | 의미 검색 + 다단계 에이전트 탐색 | 높음 |
| 이 레포와 다른 레포의 인증 설계를 비교해줘 | 구조화 메타데이터 비교 | 프로젝트 간 의미 검색 | 높음 |

### 8.3 권장 검색 파이프라인

```text
질문 분류
→ 메타데이터 필터(레포, 브랜치, 언어, 파일 유형, 역할)
→ 정적/그래프 검색 + 정확 검색/BM25 + 벡터 검색 중 필요한 방식 실행
→ 후보 병합 및 재순위화
→ 관련 파일의 원문과 주변 코드 읽기
→ 근거 충돌 검사
→ 답변 또는 문서 섹션 생성
→ 파일 경로·근거·불확실성 표시
```

### 8.4 임베딩을 생략하거나 지연할 조건

- 파일 수와 코드량이 매우 작은 레포
- 일회성 얕은 분석만 요청한 경우
- 실행법·환경변수·기술 스택처럼 결정적 파싱으로 충분한 경우
- 비용 또는 처리 시간이 사용자 가치보다 큰 경우

### 8.5 임베딩을 적극 사용할 조건

- 레포가 커서 후보 파일을 줄여야 하는 경우
- 같은 의미가 다양한 이름으로 표현된 경우
- 기능이 여러 계층과 모듈에 흩어진 경우
- 역할별 Q&A와 문서 재생성이 반복되는 경우
- 여러 레포 또는 여러 버전을 비교하는 경우

### 8.6 청킹 원칙

코드는 고정 글자 수로만 자르면 함수와 클래스 문맥이 깨질 수 있다. 가능하면 다음 단위를 사용한다.

- 함수, 클래스, 인터페이스, 라우트 핸들러
- 설정 블록과 스크립트 항목
- Markdown 제목 단위
- 파일 요약 및 디렉토리 요약

각 청크에는 `repo_id`, `commit_sha`, `path`, `language`, `symbol`, `start_line`, `end_line`, `imports`, `imported_by`, `role_tags` 같은 메타데이터를 연결한다. 의존성은 벡터에만 맡기지 않고 명시적 관계 데이터로 저장한다.

### 8.7 반드시 측정해야 할 것

임베딩 채택을 기술 취향으로 결정하지 않고 평가 데이터로 검증한다.

- 검색 정답 파일이 Top-k 안에 포함되는 비율
- 근거 파일 정확도와 누락률
- 키워드 검색 단독 대비 하이브리드 검색 개선폭
- 질문당 입력 토큰, 응답 시간, 모델 비용
- 작은·중간·큰 레포별 최초 분석 시간
- 인덱스 재사용 시 후속 질문 시간

하이브리드가 실제로 개선하지 못하면 해당 질문 유형에서는 벡터 검색을 빼는 것이 맞다.

## 9. 제품 입력 설계: 역할과 레포 사용 목적을 먼저 받는다

모든 질문을 회원가입 단계에서 강제로 받으면 진입 장벽이 높아진다. 최소 가입 후, 첫 프로젝트 등록 과정에서 2~4개의 선택으로 분석 목적을 설정하고 나중에 변경할 수 있게 한다.

### 9.1 사용자 역할

```text
개발자
├─ 프론트엔드
├─ 백엔드
├─ 풀스택
├─ 데이터/AI
└─ DevOps/인프라

협업자
├─ PM/기획
├─ 디자이너
├─ QA
└─ 교육자/리뷰어
```

역할은 계정의 고정 정체성이 아니라 프로젝트별로 다르게 선택할 수 있어야 한다. 같은 사용자가 한 프로젝트에서는 백엔드 개발자이고 다른 프로젝트에서는 리뷰어일 수 있기 때문이다.

### 9.2 레포와의 관계

| 선택지 | 사용자의 실제 목적 | 분석 결과의 강조점 |
|---|---|---|
| 실제 담당 팀 프로젝트 | 빠르게 합류하고 수정 시작 | 실행법, 담당 영역, 영향 관계, 팀 온보딩 |
| 개인·사이드 프로젝트 | 전체 구조 점검 및 재진입 | 전체 요약, 개선점, 잊은 맥락 복원 |
| 레퍼런스·학습 프로젝트 | 구현 방식 학습 | 아키텍처, 핵심 패턴, 읽을 순서, 학습 질문 |
| 오픈소스 기여 대상 | 첫 기여 준비 | 기여 가이드, 이슈 관련 영역, 테스트·빌드 |
| 인수인계·유지보수 프로젝트 | 위험 없이 운영 이해 | 배포, 설정, 외부 연동, 위험 요소, 레거시 경계 |

### 9.3 현재 목표

복수 선택을 허용한다.

- 로컬에서 실행하기
- 전체 구조 이해하기
- 특정 기능 흐름 찾기
- 첫 수정 또는 기여 준비하기
- PM·디자이너·QA 관점에서 이해하기
- 인수인계 문서 만들기
- 다른 레포와 비교 학습하기

### 9.4 사용자의 익숙함 수준

- 처음 접하는 기술 스택
- 일부 기술만 익숙함
- 기술은 익숙하지만 이 레포는 처음
- 프로젝트에 이미 참여 중

이 정보는 설명의 난이도와 용어 풀이 수준을 조절하는 데 사용한다.

### 9.5 UI 흐름 와이어프레임

```text
회원가입 → 최소 정보 (이메일, 이름)
→ 첫 프로젝트 등록 화면
  ┌──────────────────────────────────────────────┐
  │  📦 프로젝트 추가                              │
  │                                               │
  │  GitHub URL 또는 로컬 업로드                    │

  │  ○ 풀스택 개발자      ○ PM/기획                │
```

## 10. 역할별 결과 화면 설계

### 10.1 공통 탭

1. **한눈에 보기**
   - 프로젝트 목적, 기술 스택, 분석 기준 커밋, 주요 모듈
2. **실행 가이드**
   - 설치·환경변수·서비스 실행 순서·확인 상태
3. **코드 맵**
   - 폴더 구조, 진입점, 주요 관계, 기능 흐름
4. **읽을 순서**
   - 추천 파일과 이유, 역할별 학습 경로
5. **질문하기**
   - 근거 파일이 연결된 Q&A (공유/개인 토글)
6. **팀 활동**
   - 팀원들의 공유 Q&A, 온보딩 진행률, 답변 보충
7. **근거와 최신성**
   - 커밋, 분석 시각, 확인됨/추론됨/미확인 상태

### 10.2 역할별 홈 카드 예시

#### 백엔드 개발자

- 서버 진입점
- 라우터 → 서비스 → 저장소 흐름
- 인증·권한 구조
- DB 연결과 모델
- 설정 및 예외 처리
- 먼저 읽을 파일 5개

#### PM·기획자

- 주요 사용자 기능
- 기능별 화면·API·데이터 연결
- 핵심 도메인 용어
- 정책이 구현된 위치
- 개발자에게 확인할 미정 사항

#### 디자이너

- 화면 및 라우트 목록
- 레이아웃·공통 컴포넌트
- 로딩·빈 상태·오류 상태
- 디자인 시스템과 에셋 위치
- UI가 호출하는 주요 API

#### QA

- 핵심 사용자 여정
- 인증 및 권한 분기
- 오류 코드와 예외 처리
- 테스트가 있는 영역과 없는 영역
- 변경 영향이 큰 파일

### 10.3 질문 추천도 역할별로 달라져야 한다

빈 채팅창 대신 역할과 목표에 맞는 질문을 제안한다.

```text
백엔드 개발자: "인증 요청이 DB까지 가는 경로를 보여줘"
PM: "프로젝트의 핵심 사용자 기능을 화면과 API 기준으로 묶어줘"
디자이너: "공통 버튼과 폼 컴포넌트가 어디서 사용되는지 보여줘"
QA: "로그인 실패 시 처리되는 예외와 UI 상태를 정리해줘"
```

## 11. 분석 UX: 얕은 분석과 깊은 분석을 구분한다

처음부터 긴 인덱싱을 강제하지 않고 빠른 가치를 먼저 보여준다.

### 11.1 1단계: 빠른 구조 분석

목표 시간은 실제 구현 후 측정해 정한다. 초기 화면에는 예상 시간을 과장하지 않고 진행 단계를 표시한다.

- 파일 트리 및 언어 통계
- manifest·설정·README 탐지
- 실행 명령 후보
- 환경변수 키 목록
- 앱 진입점과 라우터 후보
- 기본 기술 스택

이 단계는 임베딩 없이도 가능해야 한다.

### 11.2 2단계: 심층 인덱싱 및 관계 분석

- AST 기반 심볼 추출
- import·호출·라우트·DB 관계 구성
- 코드 청킹 및 임베딩
- 파일·디렉토리 요약
- 역할별 온보딩 리포트 생성
- 심층 Q&A 준비

### 11.3 사용자에게 보여줄 상태

```text
레포 가져오는 중
→ 파일 분류 중
→ 실행·설정 정보 확인 중
→ 코드 관계 분석 중
→ 의미 검색 인덱스 생성 중
→ 역할별 리포트 생성 중
→ 분석 완료
```

"AI 분석 중" 하나로 감추지 말고, 현재 확보된 결과부터 볼 수 있게 한다. 실패한 단계가 있어도 이미 완료된 얕은 분석 결과는 유지한다.

## 14. 성공 지표

기술 지표만으로는 제품 가치를 증명할 수 없다. 사용자 결과와 검색 품질을 함께 측정한다.

### 14.1 사용자 가치 지표

- 프로젝트 등록 후 첫 유용한 결과까지 걸린 시간
- 신규 사용자가 로컬 실행 방법을 찾는 데 걸린 시간
- 첫 핵심 파일 또는 첫 수정 지점을 찾는 데 걸린 시간
- 온보딩 리포트 공유율
- 같은 프로젝트에 두 명 이상이 접근한 비율
- 팀장·시니어의 반복 설명 시간 변화
- 추천 파일과 질문 카드의 사용률

### 14.2 협업 지표 (신규)

- 팀스페이스 생성률 (프로젝트 중 팀스페이스가 있는 비율)
- 팀원 초대율 (팀스페이스 당 평균 멤버 수)
- 공유 Q&A 대비 개인 Q&A 비율
- 개인 → 공유 전환 비율
- 시니어 답변 보충 비율
- "도움이 됐어요" 반응률
- Q&A → 온보딩 리포트 승격 비율
- 온보딩 체크리스트 완료율
- 온보딩 타임라인 평균 소요 시간

### 14.3 품질 지표

- 실행 명령·환경변수·진입점 탐지 정확도
- 질문별 정답 근거 파일 Top-k 포함률
- 파일 경로 또는 줄 근거가 없는 주요 주장 비율
- `confirmed / inferred / conflict` 분류 정확도
- README와 코드 충돌 탐지 정밀도
- 역할별 결과에 대한 유용성 평가

### 14.4 비용·성능 지표

- 레포 크기별 얕은 분석과 깊은 분석 시간
- 레포당 임베딩 및 생성 비용
- 질문당 검색·생성 지연 시간
- 인덱스 재사용률과 캐시 적중률
- 증분 분석이 전체 재분석 대비 절약한 시간과 비용

## 17. 발표 및 소개 문구

### 17.1 한 줄 소개

> **CodeMap은 낯선 코드베이스에서 발생하는 "이해의 비대칭"을 해소하는 팀 협업 온보딩 플랫폼입니다.**

### 17.2 문제 설명

> README는 프로젝트마다 목적과 품질이 다르고 코드 변경과 자동으로 동기화되지 않습니다. 새 팀원은 정보가 없는 것이 아니라, 흩어진 정보 중 무엇을 믿고 어떤 순서로 읽어야 하는지 몰라 시간을 씁니다. 그리고 혼자 이해한 내용이 맞는지 확인할 방법이 없습니다.

### 17.3 코딩 에이전트와의 차이

> Codex나 Claude Code 같은 범용 코딩 에이전트도 코드베이스를 매우 잘 탐색합니다. CodeMap은 그 능력을 부정하지 않습니다. 대신 개인이 질문하고 작업하는 1인 세션을 넘어, 같은 레포를 역할별로 이해하고 근거가 남는 온보딩 결과를 팀이 함께 만들고 재사용하도록 하는 데 집중합니다.

### 17.4 RAG 사용 이유

> CodeMap은 모든 질문에 벡터 검색을 강제하지 않습니다. 실행법과 환경변수는 설정 파일을 직접 분석하고, 함수와 의존성은 AST와 관계 그래프로 추적하며, 표현이 다양한 기능 탐색에는 키워드와 임베딩 검색을 함께 사용합니다. 이후 에이전트가 원문 파일을 확인해 근거와 함께 답합니다.

### 17.5 핵심 슬로건 후보

```text
낯선 코드베이스에 들어가는 가장 빠른 지도
```

```text
질문하기 전에, 어디서부터 봐야 하는지
```

```text
혼자 읽는 코드에서, 함께 이해하는 프로젝트로
```

```text
시니어의 설명을 기다리지 않아도 되는 온보딩
```

```text
팀의 코드 이해를 동기화하는 가장 빠른 방법
```

## 18. 최종 포지셔닝 문장

> **CodeMap은 낯선 코드베이스 앞에서 발생하는 "이해의 비대칭"을 해소하는 팀 협업 온보딩 플랫폼이다. 역할별 프로젝트 맵, 근거 기반 설명, 공유 Q&A 공간을 통해 시니어의 반복 설명 없이 팀 전체가 같은 수준으로 프로젝트를 이해하도록 돕는다.**

제품의 방향을 짧게 정리하면 다음과 같다.

```text
코사인 유사도 검색 서비스가 아니라,
정확한 프로젝트 이해를 위해 여러 검색 방식을 조합하는 서비스
```

```text
범용 코딩 에이전트의 축소판이 아니라,
코딩 에이전트와 팀 사이에 놓이는 공용 코드 이해 레이어
```

```text
한 번 생성하고 잊는 AI 요약문이 아니라,
근거·역할·커밋·최신성을 가진 살아 있는 온보딩 맵
```

```text
혼자 질문하고 끝나는 1인 세션이 아니라,
팀이 함께 이해하고 검증하는 협업 온보딩 공간
```

## 19. v3에서 확정할 제품 원칙

1. CodeMap의 상위 미션은 "이해의 비대칭 해소"다.
2. 임베딩과 코사인 유사도는 USP로 내세우지 않는다.
3. 작은 레포와 결정적 질문에는 임베딩을 강제하지 않는다.
4. CodeMap의 기본 경험은 빈 채팅창이 아니라 선제적 온보딩 맵이다.
5. 핵심 타겟은 연차가 아니라 낯선 레포에 진입한 사람이다.
6. PM·디자이너·QA는 역할별 뷰의 확장 타겟으로 두되, MVP 정확도를 희생하지 않는다.
7. 모든 주요 설명은 파일 근거와 검증 상태를 가져야 한다.
8. 분석 기준 커밋을 기록하고 문서의 최신성을 관리한다.
9. 서버 분석의 비용과 보안 문제를 인정하고 팀 재사용 가치가 큰 상황에 집중한다.
10. 범용 코딩 에이전트와 대체 관계가 아니라 상호 보완 관계를 만든다.
11. 기술 선택은 데모 완성도가 아니라 검색 품질·토큰·시간·비용 평가로 결정한다.
12. 팀 협업 기능은 "코드 이해의 공유"로 범위를 한정하고, 범용 협업 도구를 재발명하지 않는다.
13. 팀 기능 없이도 1인 사용자에게 온보딩 맵·Q&A의 완전한 가치를 제공한다.

## 20. 참고한 현재 제품 설명

경쟁 제품의 범위를 지나치게 좁게 정의하지 않기 위해 공식 설명을 기준으로 확인했다.

- [OpenAI Codex](https://openai.com/codex/): 코드 이해, 구현, 리뷰, 클라우드 및 팀 워크플로를 포함하는 범용 코딩 에이전트
- [Anthropic Claude Code 개요](https://docs.anthropic.com/ko/docs/claude-code/overview): 코드베이스 탐색, 수정, 명령 실행, 외부 도구 연결을 지원하는 에이전틱 코딩 도구
- [Google Antigravity 시작하기](https://codelabs.developers.google.com/getting-started-google-antigravity): 로컬 에이전트 관리와 IDE 기반 코드 이해를 제공하는 에이전트 개발 플랫폼

이 비교의 목적은 "경쟁 제품에 없는 기능"을 단정하는 것이 아니라, 빠르게 확장되는 범용 도구와 기능 목록으로 경쟁하지 않고 CodeMap이 **온보딩이라는 반복 가능한 팀 워크플로**를 얼마나 더 잘 설계해야 하는지 분명히 하는 데 있다.
