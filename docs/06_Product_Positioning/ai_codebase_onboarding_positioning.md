# AI 코드베이스 온보딩 도우미 프로젝트 방향 정리

## 1. 프로젝트 한 줄 정의

**AI 코드베이스 온보딩 도우미**는 GitHub 레포지토리 URL을 입력하면 코드베이스 구조, 실행 방법, 핵심 파일, 읽는 순서, 주요 기능 흐름을 분석하여 **신입 개발자와 협업자를 위한 팀 공유용 온보딩 문서**를 자동 생성하는 서비스이다.

이 서비스는 Codex, Claude Code, Antigravity처럼 코드를 직접 수정하거나 디버깅하는 개발자용 코딩 에이전트가 아니라, 새 팀원이나 협업자가 프로젝트를 처음 이해할 수 있도록 **온보딩 리포트**를 생성하는 데 초점을 둔다.

---

## 2. 문제의식

기존 코드베이스에 새로 합류한 사람은 보통 다음과 같은 어려움을 겪는다.

- README가 부실하거나 오래되어 실제 실행 방법을 알기 어렵다.
- 폴더 구조만 보고는 전체 흐름을 이해하기 어렵다.
- 어떤 파일부터 읽어야 할지 알기 어렵다.
- 실행 방법, 환경변수, 주요 진입점, API 구조가 흩어져 있다.
- 시니어 개발자나 팀장이 매번 같은 설명을 반복해야 한다.
- PM, QA, 디자이너 등 비개발 협업자는 코드 에디터나 CLI 기반 AI 도구를 사용하기 부담스럽다.
- AI 코딩 도구는 질문을 잘하면 답을 잘하지만, 초보자나 비개발자는 무엇을 질문해야 할지 모르는 경우가 많다.

따라서 이 프로젝트의 핵심 문제는 단순히 “코드에 대해 질문할 수 있게 하는 것”이 아니라, **처음 레포지토리를 받은 사람이 프로젝트의 구조와 흐름을 빠르게 이해하도록 돕는 것**이다.

## 3. 타겟 사용자

### 1. 신입/주니어 개발자 온보딩

프로젝트에 처음 합류한 신입 또는 주니어 개발자는 레포지토리를 받아도 어디서부터 읽어야 할지 모르는 경우가 많다.

이 서비스는 다음과 같은 정보를 제공하여 초기 적응 시간을 줄인다.

- 프로젝트 개요
- 기술 스택
- 실행 방법
- 핵심 폴더 구조
- 처음 읽어야 할 파일
- 주요 기능 흐름
- 환경변수 및 설정 파일
- 신입 개발자가 주의해야 할 부분

---

### 2. 팀 프로젝트에 새로 합류한 팀원

이미 진행 중인 프로젝트에 중간 합류한 팀원은 전체 맥락을 파악하기 어렵다.

이 서비스는 GitHub 레포지토리 기반으로 프로젝트의 구조를 빠르게 요약하여, 새 팀원이 다음을 이해할 수 있도록 돕는다.

- 현재 프로젝트가 어떤 구조로 구성되어 있는지
- 프론트엔드와 백엔드가 어떻게 나뉘어 있는지
- 주요 기능이 어떤 파일에서 시작되는지
- 어떤 순서로 코드를 읽으면 좋은지
- 로컬 실행을 위해 무엇을 설정해야 하는지

---

### 3. 신입에게 프로젝트를 설명해야 하는 시니어/팀장

시니어 개발자나 팀장은 신입에게 프로젝트 구조를 반복적으로 설명해야 하는 부담이 있다.

이 서비스는 팀장이 매번 설명해야 하는 내용을 자동 문서화하여 다음과 같은 효과를 제공한다.

- 반복적인 온보딩 설명 시간 감소
- 신입에게 공유할 기본 문서 자동 생성
- 프로젝트 구조 설명의 표준화
- 핵심 파일과 읽는 순서 정리
- 팀 내부 인수인계 자료로 활용 가능

---

### 4. 코드 구조를 이해해야 하는 PM/QA/교육자

PM, QA, 교육자는 코드를 직접 수정하지 않더라도 프로젝트 구조를 이해해야 할 때가 있다.

이 서비스는 개발자 중심의 세부 구현보다 전체 구조와 기능 흐름을 설명하는 데 초점을 두어, 다음과 같은 사용자에게 도움을 줄 수 있다.

- PM: 기능 흐름과 API 구조 파악
- QA: 테스트해야 할 주요 기능과 흐름 이해
- 교육자: 학습자에게 프로젝트 구조 설명
- 리뷰어: 코드베이스의 전반적인 구성 파악

---

### 5. PM, 디자이너, 기획자, QA 등 비개발 협업자

비개발 협업자는 코드 에디터, 터미널, 로컬 개발환경, CLI 기반 AI 도구를 사용하는 데 부담을 느낄 수 있다.

이 서비스는 GitHub URL만 입력하면 웹에서 읽기 쉬운 문서 형태로 결과를 제공하므로, 비개발 협업자도 다음을 쉽게 파악할 수 있다.

- 프로젝트의 주요 기능
- 화면 또는 API 흐름
- 프론트엔드와 백엔드의 역할
- 데이터가 이동하는 흐름
- 개발자와 커뮤니케이션할 때 참고할 수 있는 구조적 정보

## 4. 기존 AI 코딩 도구와의 차별점

Codex, Claude Code, Antigravity 같은 도구는 개발자가 코드 수정, 디버깅, 테스트 실행, 리팩토링 등을 수행하기 위한 **개인 작업용 코딩 에이전트**에 가깝다.

반면 이 프로젝트는 코드를 직접 수정하는 것이 아니라, 프로젝트를 처음 접하는 사람이 이해할 수 있도록 **팀 공유용 온보딩 문서**를 생성하는 데 초점을 둔다.

| 구분 | Codex / Claude Code / Antigravity | AI 코드베이스 온보딩 도우미 |
|---|---|---|
| 주요 목적 | 코드 수정, 디버깅, 개발 보조 | 코드베이스 이해, 온보딩 문서 생성 |
| 사용 환경 | 로컬 개발환경, IDE, CLI | 웹 기반 서비스 |
| 주요 사용자 | 개발자 개인 | 신입 개발자, 팀원, 팀장, PM, QA, 비개발 협업자 |
| 결과물 | 대화 답변, 코드 수정, 패치 | 공유 가능한 온보딩 리포트 |
| 강점 | 실시간 코드 작업 | 구조 요약, 읽는 순서, 실행 방법 정리 |
| 포지션 | 코딩 에이전트 | 코드베이스 문서화/온보딩 도구 |

## 5. 핵심 페인포인트

이 프로젝트가 해결하려는 핵심 페인포인트는 다음과 같다.

1. **README의 한계**
   - README가 없거나 부실하다.
   - 실제 실행 방법과 문서 내용이 다를 수 있다.
   - 프로젝트 구조나 읽는 순서까지 설명하지 않는 경우가 많다.

2. **초기 코드 탐색의 어려움**
   - 파일과 폴더가 많으면 어디서부터 봐야 할지 알기 어렵다.
   - 진입점, 설정 파일, 라우터, DB 연결 파일이 흩어져 있다.

3. **반복되는 온보딩 비용**
   - 시니어나 팀장이 새 팀원에게 같은 설명을 반복해야 한다.
   - 프로젝트마다 온보딩 문서를 새로 만들기 번거롭다.

4. **비개발 협업자의 접근성 문제**
   - PM, QA, 디자이너는 로컬 개발환경을 세팅하거나 CLI 도구를 쓰기 어렵다.
   - 코드 전체를 직접 읽기보다는 구조화된 설명이 필요하다.

5. **AI 코딩 도구의 질문 의존성**
   - Codex나 Claude Code는 사용자가 질문을 잘하면 좋은 답을 준다.
   - 하지만 신입이나 비개발자는 어떤 질문을 해야 하는지부터 모를 수 있다.
   - 따라서 질문하기 전에 볼 수 있는 기본 온보딩 문서가 필요하다.

### 5. 근거 파일 기반 설명

LLM이 단순히 추측하여 설명하는 것이 아니라, 실제 레포지토리에서 찾은 파일과 코드 조각을 근거로 답변한다.

예시:

```text
인증 로직은 다음 파일에서 확인할 수 있습니다.

- backend/app/infra/security.py: JWT 생성 및 검증
- backend/app/api/auth.py: 로그인 API 라우터
- backend/app/deps.py: current_user 의존성 주입
```

이렇게 근거 파일을 함께 제시하면 결과의 신뢰성을 높일 수 있다.

## 7. RAG와 임베딩의 역할

이 프로젝트에서 RAG와 임베딩은 서비스의 목적 그 자체라기보다, **코드베이스에서 필요한 근거를 찾기 위한 내부 기술**로 활용된다.

코드 레포지토리 분석에서는 임베딩 검색이 항상 키워드 검색보다 우월한 것은 아니다.

예를 들어 다음과 같은 질문은 키워드 검색이나 정적 분석이 더 정확할 수 있다.

- 환경변수는 어디에 정의되어 있나요?
- 실행 명령어는 어디에 있나요?
- 라우터 파일은 어디에 있나요?

반면 다음과 같은 질문은 의미 기반 검색이 보완적으로 유용할 수 있다.

- 인증 로직은 어디에 있나요?
- 처음 읽어야 할 핵심 파일은 무엇인가요?
- DB 연결 흐름을 설명해주세요.
- API 요청은 어떤 흐름으로 처리되나요?

따라서 이 프로젝트는 단순한 벡터 검색만 사용하는 것이 아니라, 다음 세 가지 방식을 결합하는 것이 적합하다.

```text
정적 분석 + 키워드 검색 + 벡터 검색
```

## 8. 기술적 접근: Hybrid Codebase Retrieval

코드베이스 분석에서는 순수 RAG보다 하이브리드 접근이 더 적합하다.

### 1. 정적 분석

정적 분석은 파일 구조와 명확한 설정 정보를 파악하는 데 사용한다.

예시:

- 폴더 구조 생성
- 프레임워크 추정
- 실행 파일 탐지
- 의존성 파일 탐지
- 환경변수 파일 탐지
- 진입점 탐지
- API 라우터 후보 탐지
- DB 모델 및 설정 파일 후보 탐지

---

### 2. 키워드 검색

키워드 검색은 명확한 단어, 파일명, 함수명, 설정명을 찾는 데 유용하다.

예시 키워드:

```text
auth
login
token
jwt
oauth
session
current_user
password
middleware
security
permission
database
db
router
controller
endpoint
config
settings
env
```

---

### 3. 벡터 검색

벡터 검색은 단어가 정확히 일치하지 않아도 의미적으로 관련 있는 코드 조각을 찾는 데 사용한다.

예를 들어 `auth`라는 단어가 없어도 다음과 같은 코드 조각은 인증 흐름과 관련될 수 있다.

```text
issueAccessToken
verifyCredentials
getPrincipal
guard
sessionProvider
hashPassword
validateUser
```

이때 임베딩 기반 검색은 키워드 검색이 놓칠 수 있는 의미적 관련성을 보완한다.

---

### 4. LLM 생성

LLM은 검색된 근거를 바탕으로 사람이 읽기 쉬운 온보딩 문서를 생성한다.

LLM의 역할은 다음과 같다.

- 코드 구조를 자연어로 설명
- 신입 개발자 관점에서 읽는 순서 추천
- 파일 간 관계 요약
- 실행 방법 정리
- 주요 기능 흐름 설명
- 주의할 점과 체크리스트 생성

```text
8. 정적 분석 + 키워드 검색 + 벡터 검색
9. 섹션별 근거 파일 수집
10. LLM 기반 온보딩 문서 생성
11. 결과 화면 출력
12. Markdown/PDF/링크 형태로 공유
```

## 11. 발표에서 사용할 수 있는 설명

### 프로젝트 정의

> 저희 서비스는 Codex나 Claude Code처럼 코드를 직접 수정하는 개발자용 에이전트가 아니라, 새 팀원이 프로젝트를 처음 이해할 수 있도록 GitHub 레포지토리를 분석해 팀 공유용 온보딩 문서를 자동 생성하는 서비스입니다.

### RAG 사용 이유

> 코드 레포지토리에서는 임베딩 검색이 항상 키워드 검색보다 우월한 것은 아닙니다. 그래서 저희는 정적 분석, 키워드 검색, 벡터 검색을 결합한 하이브리드 방식을 사용했습니다. 이를 통해 레포 전체를 LLM에 그대로 넣지 않고, 섹션별로 필요한 근거 파일과 코드 조각을 선별한 뒤 온보딩 문서를 생성합니다.

### 기존 AI 코딩 도구와의 차이

> 기존 AI 코딩 도구는 개발자가 코드 수정, 디버깅, 테스트 실행을 위해 사용하는 개인 작업 도구에 가깝습니다. 반면 저희 서비스는 신입 개발자, 팀장, PM, QA, 디자이너 등 다양한 팀원이 프로젝트 구조를 이해할 수 있도록 공유 가능한 문서 형태의 결과물을 생성하는 데 초점을 둡니다.

## 12. 최종 포지셔닝

이 프로젝트는 다음과 같이 정의할 수 있다.

> **AI 코드베이스 온보딩 도우미는 GitHub 레포지토리 URL만으로 프로젝트 구조, 실행 방법, 핵심 파일, 읽는 순서, 주요 기능 흐름을 분석하여 신입 개발자와 협업자를 위한 팀 공유용 온보딩 문서를 자동 생성하는 서비스이다.**

핵심 포지션은 다음과 같다.

```text
코딩 에이전트가 아니라,
코드베이스 온보딩 문서화 도구
```

```text
개발자 개인의 실시간 코드 수정 도구가 아니라,
팀 전체가 공유할 수 있는 프로젝트 이해 도구
```

```text
단순 RAG 데모가 아니라,
정적 분석 + 키워드 검색 + 벡터 검색 + LLM 요약을 결합한 하이브리드 코드베이스 온보딩 서비스
```
