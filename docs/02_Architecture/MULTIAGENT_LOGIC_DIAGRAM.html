CodeMap AI Multi-Agent Architecture

LangGraph State 공유형 멀티에이전트 합의안 시각화

핵심 합의 같은 고성능 LLM 엔진을 공유하더라도 LangGraph 안에서는 역할별 node/agent를 분리합니다. 다만 Route Node는 LLM agent가 아니라 deterministic routing/control node로 둡니다. 카드를 클릭하면 각 역할과 회의 합의 포인트를 확인할 수 있습니다.

Team Decision
  • 단일 만능 agent에 모든 도구를 몰아주지 않는다.
  • Worker별 multi-agent 분리는 유지한다.
  • Route Node는 LLM agent로 만들지 않는다.
  • State 공유로 LLM 간 요약 릴레이를 줄인다.
  • 직렬 LLM 릴레이 대신 병렬 Worker 실행을 우선한다.

LangGraph State 공유형 하이브리드 멀티에이전트

회의의 핵심은 “LLM을 많이 붙이느냐”가 아니라, 어떤 책임을 LLM agent로 두고 어떤 책임을 일반 코드 node로 통제할지입니다. 이 구조는 LangGraph의 State, Node, Edge 모델을 사용해 LLM node와 deterministic node를 같은 graph 안에서 함께 운용합니다.

목표: Supervisor LLM 1회, Worker 병렬 실행, Final Answer LLM 1회 중심으로 직렬 지연을 줄인다.

State는 애플리케이션의 현재 상태를 나타내는 공유 데이터 구조입니다. 각 node는 LLM일 수도 있고 일반 코드일 수도 있으므로, agentic node와 deterministic workflow node를 섞는 방식이 자연스럽습니다.

State 공유의 목적은 속도 자체를 자동 해결하는 것이 아니라, 중간 LLM 요약 릴레이로 인한 정보 유실을 줄이는 것입니다.
4 LLM AGENT

Supervisor Agent

의도 분석, 오타/모호성 정리, 필요한 worker 선택, 접근 범위 plan JSON 생성. 최종 답변은 작성하지 않고 구조화된 계획만 만든다.

↓ access_plan · rewritten_query를 State에 write
5 CODE NODE

Route Node

LLM 아님. plan JSON 검증, allowlist 검사, path traversal 차단, worker 병렬 라우팅, timeout/retry/fallback 제어를 코드로 수행.

↓ 검증된 작업만 병렬 fan-out
Worker Agents: 같은 모델을 공유할 수 있지만 prompt, tool, schema, state 책임은 분리
6A LLM

Search Worker

Query rewrite, semantic search, embedding/vector search 전략

6B CODE

Dir Worker

repo 구조 탐색. 도구는 작고 제한된 단일 책임

6C CODE

Grep Worker

키워드, 정규식, alias 기반 빠른 후보 탐색

6D CODE

Read Worker

허용된 후보 파일만 읽고 line/evidence를 State에 기록

6E OPTIONAL LLM

Code Reasoning

필요 시 읽은 코드 조각을 해석. MVP에서는 생략 가능

↓ raw evidence를 State에 직접 write
7 CODE FIRST

Evidence Aggregator Node

MVP는 코드 추천. 중복 제거, 파일 경로/라인/근거 정리, token budget 압축. 고도화 시에만 LLM 압축을 선택.

8 LLM AGENT

Final Answer Agent

State의 원본 근거와 compact_context를 기반으로 최종 답변 생성. Route Node 요약본에 의존하지 않으며 chat/final_answer_agent.py에 둔다.

LANGGRAPH STATE

CodeMapState Shared Memory

모든 node가 같은 상태 객체를 읽고 쓰며, 중간 자연어 릴레이를 최소화한다.

user_query 원본 사용자 질문: “로그인 어딧음?”
rewritten_query login, signin, auth, authentication, 회원가입, register
access_plan intent, selected_workers, allowed_paths, risk_level, parallel_groups
security_result allowlist, path traversal, denied paths, fallback policy
worker_results raw snippets, file paths, line ranges, search score, tool metadata
compact_context 중복 제거와 token budget 정리 후 Final Answer에 전달할 근거 묶음
final_answer SSE로 사용자에게 보낼 근거 기반 최종 답변
데이터 흐름: Supervisor write → Route Node read/validate → Workers write raw evidence → Aggregator compact → Final Answer read.
Response Return Flow 답변이 다시 사용자에게 돌아가는 경로를 별도 레일로 표시

MEETING SUMMARY

팀에 설명할 한 문장

같은 LLM 모델을 공유하더라도 역할별 node/agent로 분리하되, Route Node는 LLM이 아니라 deterministic control node로 둔다.

Role Matrix 회의에서 합의한 “어디까지 LLM인가”를 빠르게 확인

LLM 사용

Supervisor, Search Worker, Final Answer. 필요 시 Code Reasoning Worker와 Evidence 압축.

LLM 비추천

chat/service 직접 agent화, Route Node LLM화, 보안 검증 자연어 판단.

코드가 더 잘하는 영역

경로 제한, allowlist, timeout, retry, worker fan-out, schema 정규화, 로그 기록.

과장해서 말하지 않을 점

오타/검색은 일반 로직으로도 일부 가능. LLM Rewriter는 품질과 시연 가치 때문에 채택.
