# CodeMap 팀 협업 가이드 (Contributing Guidelines)

본 문서는 원활하고 충돌 없는 팀 협업을 위해 약속된 **Git Branch 및 Commit 규칙**을 정의합니다. 모든 팀원은 작업 전 이 규칙을 숙지해 주시기 바랍니다.

---

## 1. Branch 규칙 (GitHub Flow)

본 프로젝트는 불필요한 브랜치를 최소화하고 직관적으로 관리할 수 있는 **GitHub Flow** 기반의 구조를 따릅니다.

### 🛡️ 기본 원칙

* **`main` 브랜치**: 언제든 즉시 배포 가능한 안정적인 코드가 유지되어야 하는 핵심 브랜치입니다. 절대 직접 Push하지 않습니다.
* **작업 브랜치**: `main`에서 브랜치를 분기(checkout)하여 개인 작업을 진행합니다.

### 🏷️ 브랜치 명명 규칙
>
> **포맷:** `{분류}/{기능명}`

* `feat/login-api` (새로운 기능 개발)
* `fix/header-bug` (버그 수정)
* `docs/readme` (문서 작업)
* `chore/setting` (기타 환경 설정)

---

## 2. Commit 메시지 규칙 (Conventional Commits)

노션에서 합의된 **커밋 메시지 컨벤션**을 따릅니다. 한 커밋에는 한 가지 문제만 담아 과거 추적이 용이하도록 합니다.

### 🏷️ 커밋 메시지 포맷
>
> **포맷:** `{커밋 유형}: {제목}`
> (빈 행)
> `{본문}`

* 예시:

```text
FEAT: 회원가입 기능 추가

- 회원가입 기능 추가
```

### 🛠️ 커밋 유형 (영어 대문자로 작성)

| 커밋 유형 | 의미 |
|---|---|
| `FEAT` | 새로운 기능 추가 |
| `FIX` | 버그 수정 |
| `DOCS` | 문서 수정 |
| `STYLE` | 코드 formatting, 세미콜론 누락, 코드 자체의 변경이 없는 경우 |
| `REFACTOR` | 코드 리팩토링 |
| `TEST` | 테스트 코드, 리팩토링 테스트 코드 추가 |
| `CHORE` | 패키지 매니저 수정, 그 외 기타 수정 ex) .gitignore |
| `DESIGN` | CSS 등 사용자 UI 디자인 변경 |
| `COMMENT` | 필요한 주석 추가 및 변경 |
| `RENAME` | 파일 또는 폴더 명을 수정하거나 옮기는 작업만인 경우 |
| `REMOVE` | 파일을 삭제하는 작업만 수행한 경우 |
| `!BREAKING CHANGE` | 커다란 API 변경의 경우 |
| `!HOTFIX` | 급하게 치명적인 버그를 고쳐야 하는 경우 |

### 📝 작성 상세 규칙

1. **커밋 유형**은 영어 대문자로 작성합니다.
2. 커밋 유형 이후 **제목과 본문은 한글**로 작성하여 내용이 잘 전달될 수 있도록 합니다.
3. 제목과 본문을 **빈 행으로 분리**합니다.
4. 제목 끝에는 마침표(`.`)를 붙이지 않습니다.
5. 제목은 영문 기준 50자 이내로 작성합니다.
6. 본문에는 변경한 내용과 이유를 설명합니다. (어떻게보다는 **무엇**과 **왜**를 설명)
7. 여러 항목이 있다면 글머리 기호(`-`)를 통해 가독성을 높입니다.

## 3. PR (Pull Request) 규칙

코드 병합을 위해 PR을 올릴 때, 리뷰어가 변경 사항을 쉽게 파악하고 테스트할 수 있도록 `.github/PULL_REQUEST_TEMPLATE.md` 템플릿을 제공합니다. (깃허브에서 PR 생성 시 자동 입력됨)

### 📝 PR 템플릿 항목

1. **작업 목적**: 이 코드를 작성한 배경과 이유
2. **주요 변경 사항**: 변경된 핵심 로직과 리뷰어가 점검해야 할 체크리스트
3. **리뷰 및 테스트 방법**: 본인 로컬 환경에서 테스트해 볼 수 있는 명확한 절차 및 명령어 (필수)
4. **실행 결과**: 관련 캡처 화면 (UI 또는 API 도구 화면)
5. **리뷰어에게 당부하는 점**: 특히 집중해서 봐야 할 부분

### 🚨 PR 작성 시 핵심 필수 규칙

1. **API 명세서 및 기능 단위로 쪼개기**: PR은 리뷰하기 편하도록 최대한 작은 단위로 분할해야 합니다. 예를 들어 `FEAT: PROJECT-REPO 분석 등록/상태/이벤트 스트리밍 API 구현 (API-001/003/005/006/007)` 처럼 한 번에 여러 API를 묶어서 PR을 올리지 마시고, **반드시 개별 API 또는 세부 명세서 단위별로 쪼개서**(예: `API-001` 1개 PR, `API-003` 1개 PR) 올려주세요.
2. **Reviewers 지정**: PR 생성 시 `Reviewers` 필드에 **본인을 제외한 나머지 팀원 모두**를 반드시 추가해 주세요.

## 4. Python PEP 8 컨벤션

### 💻 코드 컨벤션 (Code Convention)

* **들여쓰기**: 탭(Tab) 대신 스페이스(Space) 4칸을 사용합니다.
* **최대 줄 길이**: 모든 줄은 최대 79자로 제한하고, 독스트링 및 주석은 72자로 제한합니다.
* **빈 줄 (Blank Lines)**:
  * 최상위 함수와 클래스 정의 사이에는 두 줄(2 lines)을 띄웁니다.
  * 클래스 내부의 메서드 정의 사이에는 한 줄(1 line)을 띄웁니다.
* **Imports**:
  * `import`문은 파일의 맨 위에, 항상 별도의 줄에 작성합니다.
  * 표준 라이브러리, 서드파티, 로컬 모듈 순서로 그룹화합니다.
* **띄어쓰기 (Whitespace)**:
  * 괄호 안이나 쉼표 바로 앞에는 공백을 넣지 않습니다.
  * 대입 연산자(`=`), 비교 연산자(`==`, `<`, `>`) 주변에는 스페이스를 한 칸씩 둡니다.

### 📝 명명 규칙 (Naming Convention)

* **클래스명**: `CapWords` (PascalCase) 방식을 사용합니다. (예: `MyClass`, `UserModel`)
* **함수 및 변수명**: 소문자와 언더스코어(`_`)를 사용하는 `snake_case` 방식을 사용합니다. (예: `def calculate_total():`, `user_id = 1`)
* **상수**: 모든 글자를 대문자로 하고 언더스코어로 구분하는 `UPPER_SNAKE_CASE`를 사용합니다. (예: `MAX_OVERFLOW`, `TOTAL_COUNT`)
* **인스턴스/클래스 메서드**: 첫 번째 인자는 항상 `self` (인스턴스), `cls` (클래스)를 사용합니다.
* **비공개 속성/메서드**: 내부적으로만 쓰이는 변수나 메서드는 언더스코어 한 개(`_`)로 시작합니다.

## 5. 작업 흐름 요약 (Golden Workflow)

1. `main`에서 새로운 브랜치를 땁니다. (예: `git checkout -b feat/login`)
2. 커밋 규칙에 맞춰 코드를 작업하고 Commit 합니다.
3. **(핵심) 내 로컬 브랜치에 `main`을 Pull 당겨와서 충돌(Conflict)이 없는지 확인하고, 있다면 내 컴퓨터에서 깨끗이 해결합니다.**
4. 충돌이 완벽히 해결된 코드를 Push 하고 PR을 올립니다.

### 📅 [2026-07-07] 프로젝트 종료 후 유지보수 단계 규칙 변경 사항

> **적용 배경**: 공식 프로젝트 개발 기간 종료에 따라 개별 개선 및 기능 추가의 편의성을 극대화하기 위해 협업 및 PR 승인 규칙을 아래와 같이 변경하여 적용합니다.

* **1. PR 승인 및 머지 요건 완화**
  * **기존**: 본인을 제외한 팀원 전체의 승인이 필요한 구조
  * **변경**: 작성자 본인을 제외한 **최소 1인의 승인(Approve)**만 존재하면 `main` 브랜치로 즉시 머지(Merge)할 수 있습니다.
* **2. 리뷰어(Reviewers) 지정**
  * **기존**: 본인을 제외한 팀원 전체 필수 지정
  * **변경**: PR 생성 시 작업 내용과 연관이 깊거나 피드백을 신속히 줄 수 있는 **최소 1인 이상의 팀원**만 선택하여 지정합니다.
* **3. PR 작성 단위 유연화 및 프로젝트 문서 업데이트 규칙**
  * **기존**: 개별 API 또는 세부 명세서 단위로 극소 분할하여 PR 작성
  * **변경**: 번거로움을 덜고 작업 흐름을 유지하기 위해, 논리적으로 완결되고 독립적으로 테스트가 가능한 **기능 단위(Feature/Fix)**로 유연하게 PR을 묶어 작성할 수 있습니다.
  * **문서 수정/추가 규칙**: 프로젝트 종료 이후 코드 작성/수정에 따른 기능 명세서 업데이트 시 다음 규칙을 준수합니다.
    * **기본 형식**: 문서 최하단에 **날짜(예: `### 📅 [2026-07-07]`)**를 적어 변경 이력(로그) 작성을 시작하고, 최종적으로 **절취선(`---`)** 하나를 그어 마칩니다.
    * **변경 이력(로그) 작성 및 본문 반영**:
      * **추가**: 하단 로그 영역에 API ID와 사유를 먼저 기재한 뒤, 상위 본문에 신규 명세를 반영합니다.
      * **수정**: 하단 로그에 수정 전 원본 명세와 사유를 먼저 보존 처리한 뒤, 상위 본문에 수정을 반영합니다.
        * *참고*: 원본 명세는 상위 도메인 대제목(##)부터 복제하되, 직접 수정하지 않는 하위 영역은 '생략'으로 대체 기재할 수 있습니다.
      * **제거**: 하단 로그에 제거 직전의 원본 명세와 사유를 먼저 보존 처리한 뒤, 상위 본문에서 해당 명세를 삭제합니다.
        * *참고*: API 전체 제거 시에는 상위 도메인 대제목(##)부터 상세 영역 전체를 복제하여 기록하며, 일부 정보만 부분 제거 시에는 해당 API 식별 정보와 함께 삭제되는 부분 명세만 기록합니다.
    * **동일 날짜 처리**: 같은 날짜에 진행된 복수의 추가/수정/제거 사항은 각각 분리하지 않고, 해당 날짜 영역(동일 절취선 내부)에 ID별로 모아서 함께 기술합니다.
    * **신규 명세 생성**: 완전히 새로운 영역에 대한 기능 명세의 경우 별도 파일로 작성하되, 문서 최상단에 **작성 시작 날짜**를 명시해야 합니다.
