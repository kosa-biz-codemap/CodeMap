근거 압축
재탐색 판단
  • LLM은 계획과 최종 답변에 집중, 파일 접근·보안 검증은 결정론적 코드로 분리
  • Dispatcher - 단일 경로 탐색에 그치지 않고, 승인된 경로들을 4개의 워커(Search, Dir, Grep, Read)로 동시 병렬 실행(Fan-out)하여 탐색 속도 극대화
  • Hybrid Search - pgvector 의미 검색 + BM25/RRF 키워드 검색
  • 일부 파일 접근이나 워커 실행이 실패하더라도, 전체 파이프라인 중지없이 수집된 부분 근거(Partial Evidence)만을 통해 후속 분석을 이어 진행
  • Evaluator - 근거 충분성 판단 후 설정된 반복 한도 안에서 re-plan을 요청합니다.
  • 최종 분석 결과는 보고서 형식(Markdown, PDF 파일)으로 다운로드 가능
Session Memory
chat_conversations └─ chat_messages[]: role, content, mode, references LangGraph checkpoints └─ thread_id = sessionId CodeMapState ├─ memory_context ├─ worker_results ├─ compact_context └─ evaluator_decision

기존 1회성 상태를 넘어서 DB 대화 이력과 LangGraph checkpoint 함께 사용. 프로젝트별 `sessionId` 기준으로 이전 질문 · 답변 근거가 Planner 입력에 복원

Database Modeling

DB 모델링: 분석 작업, 코드 지식, 대화 기억

검색 성능

code_nodes.embedding에 HNSW cosine index 적용해 1536차원 임베딩 검색을 빠르게 수행

중복 방지

job_id + path + chunk_index 유니크 제약과 in-progress 부분 유니크 인덱스 통해 중복 분석 제어

복원성

채팅 메시지, references, checkpoints를 분리 저장해 UI 복원과 Agent 실행 복원을 동시 지원합니다.

users

email, hashed_password, refresh_tokens FK로 회원 인증 기준을 보존

refresh_tokens

user_id, token, expires_at 인덱스로 JWT 재발급과 로그아웃 처리

analysis_jobs

repo_url, branch, status, stage, progress, report_json, user/team 범위

source_files

repo_id, file_path, raw_code, file_summary로 원본 파일 스냅샷 저장

code_chunks

file_id, start_line, end_line, symbol, language, embedding_vector 검색 단위

file_dependencies

source_file_id와 target_file_path로 파일 import fan-in/out 구성

code_nodes

job_id, path, type, depth, chunk_index, content, summary, embedding, metadata

code_dependencies

source_id, target_id, relation 복합 PK로 코드 노드 간 의존 관계 저장

docs

repo_id, job_id, doc_type, content, version, is_active, report_json

chat_conversations

repo_id별 title, created_at, updated_at으로 대화 스레드 복원

chat_messages

conversation_id, role, content, mode, references JSONB로 근거 칩 보존

teams / checkpoints

teams, members, invites로 공유 범위 관리. checkpoints/blobs/writes는 Agent 상태 저장

채팅 UX

  • Chrome 번역으로 메시지 DOM이 변형되는 문제를 막기 위해 `notranslate`와 `translate="no"`를 적용
독립 세션 복원

  • 백엔드는 `chat_conversations`, `chat_messages`, `sessionId`, LangGraph `thread_id`로 대화 맥락을 이어감
  • 최근 PR #310, #317에서 프로젝트 전환 시 이전 대화가 남는 문제를 해결

Feature 4

Agent 방어 로직 및 파이프라인 확장

환각·무한 루프 방어
  • Evaluator - 근거 부족한 경우만 `replan_started` 발생시키며 `AGENT_MAX_REPLANS`는 0~3 범위로 제한됨
  • state에 `attempted_signatures`를 누적해 0건 반환 탐색 반복 차단
  • Dispatcher - allowlist와 path traversal 검증을 거친 plan만 worker로 전달
  • Grep Worker에는 safe regex timeout guard 적용해 ReDoS성 패턴 인한 지연 방지
파이프라인 확장
  • Clone → Parse → Embed → Report → Docs 단계로 저장소 분석 파이프라인 구성
  • AST 품질 도구, 환경 검증 도구, Hybrid Search, 문서 생성 API가 독립 도메인으로 확장
  • 온보딩 가이드 생성, 파일 단위 요약, 추천 읽기 순서, PDF 다운로드 UI까지 산출물 흐름 연결
  • 최근 배포 안정화에서는 분석 페이지 localStorage SWR 캐시 적용해 DB/API 반복 요청 의존도 감소시킴

Problem
  • Agent가 근거 부족 상황에서 같은 탐색 반복 및 None/set 병합 오류로 실패하는 문제 발생함

  • 배포 컨테이너의 `git` 누락 등 로컬 · 운영 환경 차이로 인한 Clone 실패 문제 직면함
Try
  • Evaluator 재탐색 한도, attempted signatures, safe regex 등 방어 로직 체계 고도화
  • RDB 대화 이력 및 LangGraph checkpoint의 중복 적재 일원화해 장기 기억 구조 단순화
  • 향후 PR 자동 리뷰, 변경 영향도 분석, 팀 workspace 권한 모델 고도화
최종 결론

CodeMap은 단순 코드 검색기가 아닌, 저장소 분석 통해 근거를 제시해 온보딩 산출물까지 제작하는 “검증 가능한 AI 코드베이스 안내자”를 목표로 구현되었습니다.

낯선 코드베이스에서 발생하는 이해의 비대칭을 해소하고, 시니어의 반복 설명을 줄이며, 신입의 막막함을 없애는 협업 온보딩 플랫폼을 지향합니다.

향후 과제 및 발전 방향 (Future Works)
  • GitHub PR 봇 연동: PR 생성 시, Agent가 변경 영향 파악해 자동 리뷰 코멘트 작성
  • 다중 저장소 교차 분석: MSA 환경에서 백엔드 · 프론트엔드 저장소 문맥 연결해 End-to-End 아키텍처 분석
  • 엔터프라이즈 RBAC: 팀 내 역할(Role)에 따라 민감한 코드 · 환경 변수 접근 제한하는 세밀한 보안 모델 도입
